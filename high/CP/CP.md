# 互補 CP

## 題目想法

一開始遇到這一題的時候，我嘗試過很多版本，一直在用 string 字串或 bitset 的 01 序列來解這題，但是後來都發現不是邏輯上有問題，就是時間會超過。所以後來我查了很多資料，也問了很多問題，才發現這一題有一個更適合的做法，就是 bitmask。

這是我第一次接觸到 bitmask。一開始覺得它好像是一個很新的演算法，不過在了解原理之後，我發現它其實和其他的 01 序列概念差不多，只是把整個狀態直接放進一個 long long 裡面，再利用二進位運算去處理集合，效率會高很多。

因為這一題的資料範圍很大，如果直接去比較每一組 CP 幾乎不可能完成，所以我開始思考怎麼把每一個集合壓縮成一個數字，最後便決定利用 bitmask 來表示每一組 CP。

## 解題過程

因為這一題最多會用到 38 個角色，所以使用 long long 就足以存下整個集合。我先輸入 m 和 n，接著建立一個 unordered_map 來儲存每一種集合。

這裡 unordered_map 的 key 代表整個集合對應的 bitmask，而 value 則代表有多少個 CP 具有相同的集合。這樣即使題目最後有重複的 CP，也能一次統計數量，不需要額外處理。

接著先建立 tmp，它是一個長度為 m、每一位都是 1 的二進位數字，之後會利用它來求出每個集合的互補集合。

進入迴圈後，我先讀入每一個字串 x，再把 tst 初始化成 0，之後利用它來表示目前這一組 CP 的集合。

接著開始掃描字串中的每一個字元。我利用字元的 ASCII 編碼去決定每個角色所對應的位置，大寫字母 A 到 Z 對應前面的編號，小寫字母 a 到 l 接在後面。雖然實際上不用真的操作 ASCII 編碼，只要利用字元相減就可以得到正確的位置。

找到位置之後，我利用 OR 把對應的位置設成 1。由於 OR 的特性，就算同一個字母出現很多次，例如 ABB 和 AB，最後得到的 bitmask 也會完全相同，因此自然符合題目的要求。

完成一組集合後，就把它放進 unordered_map，並將它出現的次數加一。

全部輸入完成之後，我開始遍歷 unordered_map 中的每一個集合。先利用 tmp 和目前集合做 XOR，求出它的互補集合 puzz，再利用 unordered_map 的 find 去尋找這個互補集合是否存在。

如果找得到，就代表目前這兩種集合可以組成互補 CP，因此把兩邊出現次數的乘積加入答案。

最後因為我是遍歷整個 unordered_map，所以每一組互補 CP 都會被計算兩次，例如 A 找到 B，之後 B 又會找到 A，因此最後只要把答案除以 2，就是題目的最終答案。

## 解題方法思考

一開始我一直想用 string 或 bitset 直接表示每一組 CP，不過後來發現不是處理起來很麻煩，就是效率不足，很難通過這題的大型測資。

後來接觸到 bitmask 之後，我才發現集合其實可以直接壓縮成一個整數，再利用二進位運算快速完成許多集合操作，例如利用 OR 建立集合、利用 XOR 求互補集合，都比原本的方法簡單許多。

另外，利用 unordered_map 統計每一種集合出現的次數，也成功解決了題目最後有重複 CP 的情況，不需要再額外判斷每一筆資料，大幅降低了時間複雜度。

## 複雜度分析

### 時間複雜度：

O(所有字串總長度 + n)

建立每個 CP 的 bitmask 需要掃描所有字元一次，之後再遍歷 unordered_map 並利用平均 O(1) 的查找尋找互補集合，因此整體時間複雜度為 O(所有字串總長度 + n)。

### 空間複雜度：

O(n)

unordered_map 最多儲存每一種不同的集合，因此需要 O(n) 的額外空間。

## 學習心得

在獨立完成這一題之後，我覺得自己又進步了一些。這一題讓我第一次真正學會並實際使用 bitmask，也讓我了解到它並不是一個很難理解的新東西，而是把集合利用二進位壓縮起來，再搭配位元運算去處理。

以前我一直覺得 bitset 已經很好用了，但做完這題之後，我發現 bitmask 在表示有或沒有這種狀態時，不但寫起來更方便，很多運算也能直接利用 OR、XOR 等位元運算完成，因此效率通常也會更好。

除此之外，我也學到了遇到瓶頸時，不一定代表自己的方向是對的。有時候多查一些資料、多了解不同的演算法，反而能發現更適合解決問題的方法，也讓自己的演算法工具箱又多了一個很實用的技巧。
