蒐集寶石
Description

有一個M×N的地圖，每一格的數字紀錄著寶石的數量，如果數字是 -1 代表牆壁。

有一位機器人一開始位於(r,c)的位置上且方向朝右邊，他遵循著以下規則行走。

1. 若機器人位於的格字內寶石數量為0，則機器人程式終止。

2. 機器人維護著一個分數 score，將 score 加上當前格的寶石數量，並且撿起一顆寶石。

3. 若 score 是k的倍數，則向右轉 90 度。

4. 若機器人面向的格子是牆壁或是超出邊界，則繼續向右轉 90 度直到面向的格子非牆壁或非超出邊界，並回到第 1 步。

ShowImage (1).png

例如機器人一開始在座標(2,1)且k=2，向右走兩步之後分數為3+2+3=8，由於8是2(k=2)的倍數所以向右轉90度。接下來往下走一步分數變為11，需要向右轉2次90度才不會面向牆壁或是邊界外的格子。

接下來向前走一步走到座標(2,3)，由於先前已經拿走一顆寶石，該位置的寶石數量變為2，因此分數變為13，再繼續往上走兩步到(0,3)處分數為16，由於16為2(k=2)的倍數所以向右轉90度。

向前走一格到(0,4)後需要向右轉兩次90度，回到(0,3)後由於寶石數量為0，機器人停止。過程中機器人總共撿了8顆寶石。


Input

第一行有5個正整數M,N,k,r,c

1≤M≤100

2≤N≤100

1≤k≤20

0≤r<m

0≤c<n

保證機器人初始位置不是牆壁。接下來有M行，每一行有N的數字，代表地圖的資訊。

(60%) M = 1
(40%) 無限制


Output

輸出機器人會蒐集幾個寶石。


Sample Input 1 

1 7 3 0 4
1 -1 2 1 2 1 0
Sample Output 1

5
Sample Input 2 

4 5 4 2 1
2 0 1 1 1
2 -1 0 2 -1 
0 3 2 3 0
1 1 -1 3 1
Sample Output 2

8

# 蒐集寶石

## 題目想法

一開始看到這一題，我就覺得它是一個在二維陣列上，藉由掃描目前狀態來判斷並操控機器人移動、轉向以及停止的題目。

寫到一半的時候，我停了一下，因為我在思考要用哪一種方式來實作這一題。原本我打算完全按照題目的流程，先判斷方向，等方向確定之後再進行移動。不過後來想了想，我還是決定把起點另外初始化，之後每次迴圈都先決定是否能往前走，再移動到下一格進行判斷。

我覺得這樣做不僅能讓迴圈裡面的程式顯得更清楚、邏輯更清晰，也能讓整個迴圈的功能變得更單純，只專心處理「下一步」要做的事情，因此最後選擇了這種寫法。

另外，因為題目限制寫著「無限制」，所以為了保險起見，我整題都使用 `long long` 來儲存數值。

## 解題過程

首先，我先輸入題目所給的 M、N、K、R、C。其中，我把 R 和 C 改成 nowR 和 nowC，讓自己在後面寫程式時更容易分辨目前位置。

接著建立一個 M × N 的二維陣列來儲存整張地圖，並把所有資料輸入進去。

後來我重新檢查題目時，發現題目只有保證起點不是牆壁，並沒有保證起點一定有寶石。
因此我補上了一個特判，如果起點就是 0，代表機器人一開始就會停止，所以直接輸出 0 並結束程式。

之後，我先初始化 score 和 rock。由於我把起點獨立處理，所以 score 一開始就是起點的寶石數，
而 rock 則代表已經拿走了一顆寶石，因此設為 1，並把起點的寶石數量減一。

方向的部分，我使用數字來表示：

- 向右：0
- 向下：1
- 向左：2
- 向上：3

也就是以向右為 0，依照順時針方向遞增。

接著便進入迴圈。因為題目保證機器人最後一定會停止，所以我直接使用無限迴圈，並在遇到寶石數量為 `0` 時結束。

在每一次迴圈中，我先建立 newR 和 newC 作為新的位置，而不是直接修改 nowR 和 nowC。這樣做是因為前方可能是牆壁或超出邊界，如果直接修改目前的位置，還需要再退回來，反而會讓程式更麻煩。

接著依照目前面向的方向，模擬往前走一步。如果這一步不合法，例如撞到牆壁或超出地圖，就讓方向向右轉，再重新回到迴圈判斷。

如果可以前進，就更新目前的位置，並檢查新的格子是否已經沒有寶石。如果是 0，就直接停止；
否則便把目前格子的寶石加入分數、取走一顆寶石，並將該格子的數值減一。

最後，再判斷目前分數是否為 K 的倍數。如果是，就將方向向右旋轉 90 度。


## 解題方法思考

這一題讓我思考最久的地方，不是在演算法，而是在程式架構。

一開始我原本想完全依照題目的流程去模擬，每一步都照著題目描述依序完成。
不過後來我發現，如果把起點另外初始化，之後每一次迴圈都只負責處理「下一步」，整體邏輯會更清楚，也更容易閱讀。

除此之外，這題還讓我發現一個自己容易犯的小錯誤。

當時我一直找不到為什麼機器人不會在分數是 `K` 的倍數時轉彎，檢查了一陣子才發現，我把

!(score % k)

寫成了

!score % k


忘記了 `!`、`%` 和比較運算子的優先順序，導致判斷完全不同。這也提醒我之後遇到需要搭配 `!` 使用的條件判斷時，要更加注意括號以及運算子的優先順序。

## 複雜度分析

時間複雜度：O(M*N + T)

其中 O(M*N) 為輸入並建立整張地圖所需的時間，而 O(T)為模擬機器人移動的時間，其中T為機器人實際撿取的寶石總數。
每次撿取寶石只會進行固定次數的判斷與更新，因此模擬的時間與撿取寶石的數量成正比。

**空間複雜度：** `O(M × N)`

需要使用一個二維陣列來儲存整張地圖，因此空間複雜度與地圖大小成正比。

## 學習心得

我覺得這一題考得算滿全面的。

它本身是一道二維陣列的題目，但並不是單純考二維陣列的操作，而是把方向維護、狀態更新、條件判斷以及模擬流程全部融合在一起，讓我必須把很多平常學過的觀念整合起來。

除此之外，我也發現程式的架構其實沒有唯一的寫法。雖然我最後沒有完全照著題目的流程去實作，而是把起點獨立初始化，讓每次迴圈都只專心處理下一步，但最後同樣能完成題目的要求，而且我自己也覺得這樣的程式比較清楚。

整體來說，這題讓我覺得很驚豔。它不只是考二維資料結構，而是把各種程式設計中常見的觀念融合在一起，真正考驗學生是否能將學過的知識靈活地應用在同一個題目中。
