特殊位置
Description

給定一個n×m的二維矩陣 a，設 x = a[i][j]，離(i,j)曼哈頓距離為x內的點數值總和 % 10 恰為x的稱之為特殊位置。定義兩個點 (a, b) 和 (c, d) 的曼哈頓距離為 |a - c| + |b - d|

請寫一個程式，輸出共有幾個特殊位置，並按照字典序由小到大輸出這些位置的座標。

子問題一 (60%) n=1
子問題二 (40%)n≤50,m≤50


Input

第一行輸入兩個正整數n,m(1≤n,m≤50)，接下來有n行，每行有m個數字，每一個數字介於0到9。


Output

第一行輸出共有幾個特殊位置，接下來輸出k行，每一行輸出兩個正整數代表作標點位。特殊位置請按照字典順序由小到大輸出。


Sample Input 1 

1 8
1 2 3 4 5 6 7 8
Sample Output 1

1
0 5
Sample Input 2 

2 3
5 2 3
4 5 6
Sample Output 2

2
0 0 
1 1


# 特殊位置

## 題目想法

一開始看到這一題，我就認定它是一個在二維陣列中進行掃圖的題目。一開始我也有想過要不要用 BFS，不過仔細看了題目的限制後，
發現 n ≤ 50m ≤ 50，整張地圖最多只有 2500 個位置，
在這麼小的範圍內，直接掃描整張地圖反而比使用 BFS 更直接，也更容易實作，因此最後決定採用掃圖的方式完成。

原本我打算使用二維陣列 vector<vector<int>> 來存放整張地圖，不過後來覺得這一題的資料範圍很小，而且如果全部都使用二維陣列來存取，整個程式看起來會比較擁擠，所以最後決定改用 `map<pair<int,int>, int>` 來儲存每個位置的數值。由於每個位置的值只有 0~9，而且整張地圖最多只有 2500 個位置，因此整個計算過程使用 `int` 就已經足夠了。

## 解題過程

一開始先輸入題目所給的 n 和 m，接著建立一個 map<pair<int,int>, int> 作為整張地圖 grid，並將所有位置的數值依序輸入。

之後再建立一個 vector<pair<int,int>> ans，用來儲存所有符合條件的特殊位置。

接著開始遍歷整張地圖，依序把每一個位置都當作目前要判斷的座標 (i, j)。每次判斷時，我都會先建立一個變數 test，用來累加符合條件的位置數值總和。

接著再次遍歷整張地圖，我這裡使用範圍型 `for` 搭配 `&` 參考來遍歷 `map`，避免每次都複製 `pair` 與數值，減少不必要的複製成本。

遍歷時先取出目前位置的 row 與 col，接著利用曼哈頓距離公式

|i-r| + |j-c|

判斷目前這個位置與 (i, j) 的距離是否小於等於 grid[{i,j}]。如果符合條件，就把目前位置的數值加入 test。

當整張地圖都遍歷完成後，再判斷 test % 10 是否等於目前位置的值。如果相等，就代表這個位置符合題目所定義的特殊位置，因此把它加入 ans。

由於我是按照 i、j 的順序依序掃描整張地圖，因此加入 ans 的順序本身就已經符合題目要求的字典序，不需要再額外排序。

最後依照題目要求，先輸出特殊位置的數量，再依序輸出 ans 中所有座標即可。

## 解題方法思考

我一開始有思考是否需要使用 BFS，不過後來發現這題其實沒有搜尋路徑或逐層擴展的需求，而只是需要判斷每一個位置周圍曼哈頓距離內所有點的數值總和，因此直接掃描整張地圖反而更加直觀。

另外，在儲存地圖時，我也考慮過使用二維陣列 vector，不過最後還是選擇使用 map<pair<int,int>, int>。雖然二維陣列存取速度會比較快，不過由於這題資料範圍非常小，因此效能上的差異幾乎可以忽略。我最後選擇自己比較習慣的寫法，也讓整體程式閱讀起來更符合自己的風格。

整體來說，這題並不需要使用特別複雜的演算法，只要依照題意完整模擬即可完成。

## 複雜度分析

**時間複雜度：**

O(n²m²)

需要先遍歷整張地圖的每一個位置，再對每一個位置重新掃描一次整張地圖，因此總共會進行兩層矩陣遍歷，時間複雜度為 O((nm)²)，也就是 O(n²m²)。

**空間複雜度：**

O(nm)

主要使用 `map` 儲存整張地圖，以及 ans 儲存所有特殊位置，因此所需額外空間與地圖大小成正比，空間複雜度為 O(nm)。

## 學習心得

我覺得這是一道很標準的二維陣列模擬與掃描題目，主要是在訓練如何遍歷整張地圖、判斷曼哈頓距離，以及利用 `pair` 來表示座標。

一開始我有思考是否需要使用 BFS，不過在分析題目的資料範圍之後，我發現並不是每一題跟搜尋有關的題目都一定要使用 BFS。有時候資料範圍很小，直接模擬反而會更簡單，也更容易維護程式。

另外，我也再次體會到，演算法的選擇除了考慮時間複雜度之外，也要根據題目的限制來決定。有時候直接掃圖就是最適合的方法，而不是刻意使用比較複雜的演算法。
