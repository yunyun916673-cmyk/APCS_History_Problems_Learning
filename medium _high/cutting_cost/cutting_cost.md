切割費用

題目敘述

有一根長度為 L 的棍子，你會把這根棍子切割 n 次。

假設一開始棍子的左端在數線上的 0 位置，右端在數線上的 L 位置。

每次切割會給定一個介於 0 到 L 之間的數字 x，表示要在位置 x 將棍子切成兩段。

每一次切割所需的費用，等於當下被切割的那一段棍子的長度。

請計算依照指定順序切割後，總共需要花費多少費用。


輸入說明

第一行有兩個整數：

n L

代表：

n：總共需要切割的次數

L：棍子的總長度


接下來有 n 行，每行有兩個整數：

x i

代表：

x：第 i 次切割的位置

i：這個切割發生的順序


保證：

1 <= i <= n

所有 i 不重複

每個切割位置皆介於 0 到 L 之間



輸出說明

輸出一個整數，代表所有切割所需的總費用。

答案可能超過 2^31，但不會超過 2^60。


配分

20 分

1 <= n <= 1000
1 <= L <= 10^7

30 分

1 <= n <= 50000
1 <= L <= 10^7

50 分

1 <= n <= 200000
1 <= L <= 10^7

範例輸入

3 7
2 2
3 1
5 3

範例輸出

14


範例說明

棍子長度為 7。

切割順序：

1. 第 1 刀切在位置 3



目前棍子：

0 -------- 3 -------- 7

費用：

7

2. 第 2 刀切在位置 2



切割的是：

0 -------- 3

長度：

3

費用：

3


---

3. 第 3 刀切在位置 5



切割的是：

3 -------- 7

長度：

4

費用：

4


總費用：

7 + 3 + 4 = 14

因此輸出：

14

# 切割費用

## 題目想法

一開始看到這一題的時候，我第一個想到的方法是建立一條數線，利用 0 和 1 來表示目前的位置是否有木棒存在。

我原本想透過這條數線直接模擬每一次切割，並且判斷目前切割位置左右兩邊的範圍。

但是後來在實際撰寫的過程中，我發現一個問題：如果要判斷某一次切割所屬木棒的左界與右界，就必須在數線上尋找最近的切口，這樣不但實作麻煩，也會造成不必要的運算。

後來我想到，其實真正有用的資訊並不是整條數線，而是之前已經切過的位置。

因為每一次切割時，只需要知道目前切割點左邊最近的切口，以及右邊最近的切口，就能知道這次被切割的木棒長度。

因此我捨棄原本建立整條數線的方法，改成利用 multiset 維護所有已經切割的位置。

這題因為測資範圍較大，所以我使用 long long 儲存所有數值，避免計算總費用時超過 int 的範圍。

## 解題過程

首先建立變數 n 和 L，分別代表切割次數以及木棒長度。

接著建立：

multiset<ll> wascut;

用來儲存目前所有已經切過的位置。

這裡我沒有直接把木棒左右邊界 0 和 L 也加入 wascut 中，原因是如果加入之後，在使用 iterator 尋找左右界時，需要額外處理 begin() 前一個位置的情況，反而會讓程式判斷變得更加複雜。

另外建立：

cost

用來記錄目前累積的切割費用。

接著我建立一個 cutting 函式，負責處理一次切割。

在函式中，我會利用：

lower_bound()
upper_bound()

找到目前切割位置左右兩側最近的切口。

如果左邊沒有其他切口，代表左界就是 0；

如果右邊沒有其他切口，代表右界就是 L。

找到左右界後：

right - left

就是這一次切割需要支付的費用。

計算完成後，再把這一次切割的位置加入：

wascut

讓後續切割可以繼續利用。

進入主函式後，我先輸入 n 和 L。

因為題目給定的是切割順序，而不是輸入順序，所以我建立：

vector<ll> org_cut;

用來按照切割順序儲存每一次切割的位置。

讀取完資料後，再利用迴圈依序執行每一次切割。

最後輸出累積完成的：

cost


## 解題方法思考

這題本身並不是程式碼很難寫，而是需要找到適合的資料結構。

如果直接建立整條數線，雖然可以模擬題目，但是當 L 很大時，不但需要大量空間，也沒有必要保存所有位置。

後來我發現，每一次切割真正需要的資訊只有：

左邊最近的切口

右邊最近的切口


因此使用 multiset 維護已切割的位置會更加適合。

multiset 會自動排序，並且提供 lower_bound() 和 upper_bound()，可以快速找到需要的位置。

這也讓我了解到，很多題目並不是單純把題目中的情境直接模擬出來，而是需要先分析：

哪些資訊是真正需要維護的。

一旦找到正確的方法，程式的實作其實並不會太困難。


## 複雜度分析

### 時間複雜度：

O(n log n)

原因：

程式需要進行 n 次切割，每一次切割中都需要使用 multiset 的：

lower_bound()

upper_bound()

insert()


這些操作皆需要 O(log n) 的時間，因此總時間複雜度為 O(n log n)。

### 空間複雜度：

O(n)

原因：

程式使用 multiset 儲存最多 n 個切割位置，以及使用 vector 儲存切割順序，因此所需空間與切割數量成正比。


## 學習心得

這題讓我學到，解題時不一定要直接按照題目表面的方式模擬。

一開始我認為需要建立整條數線，但後來發現那只是保存了大量不必要的資訊。

真正重要的是找到每次切割時需要知道的內容，而這題只需要維護已經切過的位置。

透過這題，我更加理解到資料結構的選擇會大幅影響解題方式。

當選擇適合的資料結構後，原本看起來需要大量模擬的問題，也可以轉換成較簡單且有效率的做法。
